# Architecture

## Overview

img2sample is a JUCE C++ audio plugin (VST3/AU/Standalone) whose UI is a WebView
(HTML/CSS/JS). Audio is generated by Google's **Lyria** models through the Gemini
API. There is **no local server and no Python backend** — the plugin talks to the
Gemini endpoint directly over HTTPS.

The user supplies a Gemini API key, drops an image (+ optional prompt / BPM / key
/ seed), and the plugin returns a short audio sample that can be auditioned,
saved, or dragged onto a DAW track.

## Components

### Plugin (C++ / JUCE)

- **Framework**: JUCE 8
- **Formats**: VST3, Audio Unit, Standalone
- **UI**: `juce::WebBrowserComponent` rendering a bundled HTML/JS app
- **Role**: host the WebView, bridge JS↔C++, call the Lyria API, decode audio,
  play it back, and provide native drag-to-DAW

### Web UI

- `plugin/Resources/index.html` — the plugin UI, talks to C++ via the JUCE bridge
- `plugin/Resources/bridge.js` — thin wrapper over `window.__JUCE__`
- `web/index.html` — a standalone browser build that calls Gemini directly via
  `fetch` and has no native bridge / drag-to-DAW

The web resources are copied into each artefact's `Contents/Resources/` at build
time (post-build step in `CMakeLists.txt`).

## Request flow (plugin)

```
JS (index.html)
  -> callNative('generateAudio', imageB64, mime, prompt, clipMode, configJson)
       via window.__JUCE__ emitEvent('__juce__invoke', ...)
  -> C++ WebBridge::generateAudio
       -> LyriaClient::generateAsync  (HTTPS POST to generativelanguage.googleapis.com)
       <- { audioData (base64), mimeType }
  -> completion resolves the JS promise (player draws waveform)
  -> WebBridge::ingestAudio writes a temp WAV for drag-to-DAW + sets the processor buffer
```

Models: `lyria-3-clip-preview` (clip mode) and `lyria-3-pro-preview` (full track).

## Bridge functions (JS → C++)

Registered in `PluginEditor::createWebViewOptions` via `withNativeFunction`:

| Function | Purpose |
|----------|---------|
| `generateAudio` | Call Lyria with image + prompt + config |
| `getApiKey` / `setApiKey` | Load/store the Gemini key locally |
| `startPlayback` / `stopPlayback` / `seekTo` | Drive in-DAW playback |
| `getPlaybackPosition` / `getIsPlaying` | Player state polling |
| `setGeneratedAudio` | Push base64 audio into the processor + temp WAV |
| `saveToDisk` / `saveFileDialog` | Export the current sample as .wav |
| `dragToDAW` | (Validity check; the actual drag is the native overlay) |
| `updateDragBtnRect` | Report the JS drag-button rect so the overlay can align |

## Audio decoding

Lyria's default output is **headerless 16-bit PCM** (`audio/L16;codec=pcm;rate=24000`),
which JUCE's `createReaderFor` cannot parse. `WebBridge::ingestAudio` tries a real
container first (WAV/MP3), and on failure falls back to `decodeRawPcm` (parses the
sample rate from the mime, assumes mono 16-bit LE), then writes a proper WAV temp
file used both for native playback and drag-to-DAW.

## Drag-to-DAW (macOS)

The WebView (WKWebView) swallows mouse events, so drag-out can't originate from JS
alone. `NativeDragView` (`UI/NativeDragView.mm`) adds a transparent `NSView` overlay
positioned exactly over the HTML "drag to DAW" button:

- JS reports the button rect via `updateDragBtnRect`; `PluginEditor` forwards it to
  `NativeDragView::setBounds`.
- The overlay's own `mouseDragged:` starts a real `NSDraggingSession` with the temp
  WAV's file URL.

Design notes (hard-won):

- **Coordinate flip**: JUCE's editor `NSView` is *flipped* (top-left origin). The
  overlay must place its frame using the raw JS y when `parentView.isFlipped`, and
  only flip (`parentH - y - h`) otherwise. Getting this wrong puts the overlay
  hundreds of pixels off and it never receives the mouse.
- **No ARC**: this target builds without `-fobjc-arc`. The mouse-down `NSEvent`
  must be stored in a *retained* property, or it deallocates before `mouseDragged:`
  uses it (use-after-free crash in the host).

## Runtime storage

| What | Where |
|------|-------|
| Gemini API key | JUCE `ApplicationProperties` ("img2sample.settings"); `localStorage` in the web build |
| Temp WAV for drag | `$TMPDIR/img2sample/sample_<ts>.wav` |

## Key design decisions

- **No backend**: generation is a direct client call to Gemini, so there is no
  server lifecycle, no model download, no venv — just an API key.
- **WebView UI**: the entire interface is HTML/JS, shared (mostly) between the
  plugin and a standalone web build.
- **Native overlay for drag**: the only reliable way to start a file drag out of a
  WKWebView on macOS.

## Known limitations

- macOS / Apple Silicon only; ad-hoc signed (Gatekeeper bypass needed).
- Requires the user's own Gemini API key and network access.
- Drag-to-DAW reliability can vary by host DAW.
